# Scope Guard

Защита минимального scope playground. Rule `scope-minimal` — always-on; этот skill — **workflow принятия решений** перед реализацией.

## Когда применять

- Запрос звучит как целое приложение: «todo с auth и API», «админка», «блог»
- Агент собирается добавить роутер, Redux/Zustand, UI-kit, Tailwind, ESLint, тесты
- Запрос затрагивает 5+ несвязанных областей за один раз
- Пользователь не уточнил, что именно нужно в первую очередь

## Алгоритм

### 1. Выдели MVP

Сформулируй **минимальную** версию, которая закрывает суть запроса:

| Запрос пользователя | MVP для playground |
|---------------------|-------------------|
| Todo-app с API | Локальный список в `useState`, без бэкенда |
| Авторизация | Один экран-заглушка или флаг «залогинен» в state |
| Несколько страниц | Условный рендер в `App.tsx`, без react-router |
| Форма с валидацией | Нативная валидация + простой state, без react-hook-form |

### 2. Разбей на итерации

Предложи 2–3 шага, каждый — отдельная небольшая задача:

```
Итерация 1 (сейчас): UI списка + добавление/удаление
Итерация 2: фильтр и localStorage
Итерация 3: API (только если попросят)
```

Реализуй **только итерацию 1**, если пользователь не выбрал иное.

### 3. Спроси, если неясно

Используй `AskQuestion` или короткий вопрос в чате, когда выбор влияет на scope:

- Нужен ли бэкенд / API или достаточно локального state?
- Одна страница или несколько (без роутера — через tabs/условный рендер)?
- Нужна ли персистентность (localStorage)?

Не задавай больше 1–2 вопросов за раз.

### 4. Явно отклоняй лишнее

Если в запросе есть неявные требования — назови их и **не делай** без подтверждения:

- react-router / TanStack Router
- Redux, Zustand, Jotai
- Tailwind, MUI, shadcn
- Vitest, Testing Library, Playwright
- FSD, feature-sliced, DI-контейнеры
- Отдельные папки `types/`, `api/`, `store/` при маленьком проекте

## Красные флаги в diff

Остановись и пересмотри scope, если diff включает:

- `npm install` чего-то из списка выше без запроса
- Новые top-level папки (`features/`, `entities/`, `shared/`)
- Правки `vite.config.ts` / `tsconfig` «на будущее»
- README, тесты, CI — без запроса

## Формат ответа пользователю

Когда сужаешь scope, кратко напиши:

1. **Что сделаю сейчас** (MVP, 1–3 пункта)
2. **Что отложу** (явный список)
3. **Вопрос** (если нужен выбор) — один, конкретный

Пример:

> Сделаю сейчас: список задач с добавлением и удалением, state в компоненте, без API.
> Отложу: авторизацию, роутер, сохранение на сервер.
> Нужно ли сохранение в localStorage?

## После согласования

Переходи к реализации через skill `add-react-feature` (или `component-from-mockup`, если есть макет).
